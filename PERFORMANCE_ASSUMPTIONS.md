# 성능 가정 (초안)

이 문서는 부하 테스트와 용량 계획을 시작할 수 있도록 초기 성능 가정을
명시적으로 정리한 것입니다. 실제 트래픽이 쌓이면 수치는 조정합니다.

작성일: 2026-04-13

## 1) 도메인 요약 (현재 코드 기준)
이 서비스는 한강공원 주차장 보조 서비스이며 구성은 아래와 같습니다.
- 내부 관리자용 웹(Flutter Web): 주차장/공지 관리
- API 서버(FastAPI): 주차장/실시간 가용(크롤러)/시설/인증/사용자 API 제공
- iOS 앱(Swift): API를 사용해 주차 상태, 공지사항 표시

확인된 주요 API 엔드포인트:
- GET /parkinglot
- GET /parkinglot/live
- POST /parkinglot
- /facility, /v1/auth, /user, /admin, /health

## 2) 트래픽 가정 (초기 시작점)
첫 부하 테스트를 위해 가정한 값입니다. 다른 목표가 있다면
아래 입력값과 계산만 바꾸면 됩니다.

가정 A (보수적 파일럿):
- DAU: 1,000
- 피크 동시접속: DAU의 5% => 50명
- 활성 사용자당 평균 요청: 10초에 1회
- 피크 RPS: 50 / 10 = 5 RPS

가정 B (소규모 런칭):
- DAU: 10,000
- 피크 동시접속: DAU의 8% => 800명
- 활성 사용자당 평균 요청: 8초에 1회
- 피크 RPS: 800 / 8 = 100 RPS

위 수치는 "피크" 기준이며, 평균 RPS는 더 낮습니다.

## 3) 엔드포인트 비율 가정
현재 기능을 기준으로 트래픽 구성은 다음과 같습니다.
- 읽기 요청: 약 85% (GET /parkinglot, /parkinglot/live, /facility)
- 쓰기 요청: 약 15% (인증, 회원가입, 관리자 업데이트)

크롤러 관련:
- /parkinglot/live는 외부 크롤링에 의존하므로
  순수 DB 조회보다 불안정하고 느립니다. 별도 측정 및 캐싱 권장.

## 4) 초기 서버 사양 가정 (API)
목표: 피크 100 RPS에서 읽기 엔드포인트 p95 < 500ms.

권장 시작 사양:
- API: 2 vCPU / 4-8 GB RAM (1대)
- DB: 별도 인스턴스 또는 매니지드 DB (2 vCPU / 4-8 GB RAM)
- 크롤러가 요청마다 실행된다면 워커 분리 또는 30-60초 캐싱 권장

피크 시 CPU가 70%를 넘으면 API를 2대로 확장하고 로드밸런서를 둡니다.

## 5) 부하 테스트 계획 (1차)
도구: k6 또는 Locust.

시나리오:
1. 기본 조회: GET /parkinglot (60%), GET /facility (20%), GET /parkinglot/live (20%)
2. 인증: 로그인/회원가입 5-10%
3. 관리자: 낮은 빈도의 쓰기 요청(1-2 RPS)

목표:
- 5 RPS (파일럿) 10분
- 50 RPS 10분
- 100 RPS 10분

측정 지표:
- 엔드포인트별 p50/p95/p99
- 에러율 < 1%
- API + DB의 CPU/RAM 사용량

## 6) 확인이 필요한 항목
- 실제 DAU 목표 및 피크 동시접속 비율
- /parkinglot/live 캐싱 여부 (권장)
- DB 종류 및 인덱스 (parkinglotname, parkinglot)
- 관리자 트래픽이 내부 전용인지 여부

DAU와 피크 동시접속을 확정하면 수치를 구체화하고 2차 계획을 추가합니다.
